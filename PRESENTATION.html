Unique Ability #1

The Locator Learning System

  • 🧠
    Learns your team's conventions
    The POM Locator Miner scans existing test POMs and records which locator strategies your team already uses — data-testid, role, xpath, Kendo patterns, Material patterns, fallback chains.
  • 📂
    Multi-run accumulation — no data lost
    Every miner run adds a new JSON file. Stage 3 merges all historical files using a time-decay weighting (newest = 1.0, older = 0.8×). The longer you use it, the smarter it gets.
  • 🛡️
    Memory-safe by design
    Files are processed one-at-a-time. Only compact aggregate statistics are kept in memory — never raw locator arrays. Capped at maxFiles (default 6) to bound merge time.
  • 🔗
    Fallback chains on every field
    Each captured control gets a candidateLocators[] ordered by the team's learned preferences. When one locator breaks in CI, Stage 6 rotates to the next — no re-crawl needed.
Unique Abilities

What No Other Tool Does

🔀
Conditional Field Probing
Stage 3 cycles through every dropdown value and radio option, re-capturing the DOM after each change. Conditional fields appear in the test plan — not as a surprise in production.
🧩
UI Library Intelligence
Kendo dropdowns, Material selects, date pickers — AutoTestGen captures the exact interaction contract per widget type. No more "selection silently did nothing" failures.
🔄
Single-Route Mode
Added a new page this sprint? Run in single-route mode. Stage 3 crawls only that URL and merges into the existing application map. Full pipeline in minutes, not weeks.
🗂️
Multi-Framework Output
Same pipeline, five frameworks. Teams migrating from Selenium Java to Playwright TypeScript get both sets of artifacts simultaneously, enabling a phased switchover.
🔒
Secrets-Safe Architecture
.env credentials are resolved in-memory and never written to generated files, output/, or git. OIDC tokens aren't stored — only the browser storage state needed for crawling.
📈
Coverage Gap Report
Stage 8 compares generated specs against the full application map. Prioritized gap report surfaces uncovered paths before your release — not after.

Developer Experience

Run It In 5 Steps

1
Copy pipeline.config.template.json → pipeline.config.json and fill in your app URL + auth
2
Run /utility-pom-locator-miner in Copilot Chat to teach AutoTestGen your locator conventions
3
Start the Playwright MCP server: npx @playwright/mcp@latest
4
Run /master-orchestrator in Copilot Chat — all 8 stages execute in sequence
5
Review output/coverage-gap-report.json and the generated POMs + specs in your project

For a single new page: set pipeline.mode: "single-route" — done in minutes instead of days.

    Run AutoTestGen on 2–3 priority applications. One SDET per app, full-mode. Validate generated POMs + specs against manual review. Measure time saved.
  • 📊
    Phase 2 — Scale (Month 2–3)
    Roll out across the migration backlog. Enable the locator miner on all existing projects. Establish a monthly miner refresh cadence. SDETs focus on test logic instead of locators.
  • 🔗
    Phase 3 — CI Integration (Month 4+)
    Hook single-route mode into the CI pipeline. When a PR touches a page component, AutoTestGen auto-updates its POM + spec. Gap report blocks merges with uncovered critical paths.

    Continuous Learning
    Each miner run enriches the locator learning index. With every sprint, AutoTestGen becomes more accurate to your team's style — the value compounds over time.
