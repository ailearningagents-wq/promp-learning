1. Overview & Architecture

AutoTestGen is an AI-agent pipeline that converts any authenticated web application into a full, production-quality test project — with Page Object Models, data factories, and spec files — without any manual locator hunting or boilerplate writing.

It supports Playwright TypeScript, Playwright JavaScript, Selenium Java, Selenium C#/.NET, selenium-js, and WebdriverIO out of the box.

Key Design Principles

  • Page-wise streaming: Stage 3 writes each page to disk before moving to the next — no full DOM accumulation in memory.
  • Resumable: Every stage checkpoints to output/pipeline-state.json; a crashed run continues from the last completed route.
  • Learning system: The POM Locator Miner teaches Stage 3 the team's locator habits across multiple runs using a memory-safe, time-decay merge.
  • Secrets-safe: .env values are resolved in memory only — never written to output/ or any generated file.

3. Quick Start (5 Minutes)

  1. Clone or open the AutoTestGen workspace in VS Code.
  2. Copy pipeline.config.template.json → pipeline.config.json in the workspace root.
  3. Copy .env.template → .env and fill in all required values (app URL, auth credentials, project path).
  4. (One-time) Install local packages: cd mcp-config/autotestgen-mcp && npm install.
    This installs playwright and @playwright/mcp into node_modules/. No browser is downloaded — the pipeline uses your system Chrome or Edge.
  5. Ensure Google Chrome or Microsoft Edge is installed on the machine. Chrome is used automatically; Edge is the fallback if Chrome is not found.
  6. (Optional but recommended) Run the POM Locator Miner first if you have an existing test project: type /utility-pom-locator-miner in Copilot Chat.
  7. In Copilot Chat, open orchestrator/master-orchestrator.prompt.md and type /master-orchestrator.
  8. The orchestrator runs all 8 stages and produces a complete test project at your configured project.existingProjectPath or project.newProjectPath.
Tip: For a new route only, set pipeline.mode: "single-route" and pipeline.singleRoute: "/your/route" in pipeline.config.json. This runs the full pipeline scoped to one page and merges results into the existing project.

5. The 8-Stage Pipeline

Stage 0 — Input Collection & Output Bootstrap

Prompt: prompts/00-input-collection.prompt.md

Creates the output/ directory structure and seeds output/pipeline-state.json. If state already exists (resume), only updates lastUpdated. No browser interaction.

Output: output/logs/stage0.log, output/pipeline-state.json

Stage 1 — Project Fingerprint

Prompt: prompts/01-project-fingerprint.prompt.md

Scans the existing project (or template) to determine tech stack, POM folder structure, test file patterns, UI libraries in use, and base class conventions.

Output: output/project-fingerprint.json

Stage 2 — OIDC Authentication

Prompt: prompts/02-oidc-auth.prompt.md

Performs the full OIDC / Ping / basic login flow via Playwright MCP. Saves the authenticated browser storage state. All subsequent crawl and generation stages reuse this state without re-authenticating.

Output: output/auth/storageState.json

Supported auth types: OIDC Ping PKCE, OIDC Ping (implicit), OIDC Standard, HTTP Basic, None.

Stage 3 — DOM Crawl & Application Discovery

Prompt: prompts/03-dom-crawl.prompt.md

The core discovery stage. Crawls all application routes, captures full DOM structure per page (forms, buttons, tables, modals, validation containers, hidden/conditional fields), and probes conditional logic.

Memory-Safety Rules (REQUIRED)

  • After capturing each page, immediately write it to output/application-map.json and update pipeline-state.json.
  • Release per-page DOM data before advancing to the next route. Only the route queue + counters persist across iterations.
  • Never accumulate full DOM snapshots for all pages in memory.
  • If crawl stalls mid-run, the next invocation resumes from stage3.lastCompletedRoute.

Locator-Learning Bias

If learning/INDEX.json exists, Stage 3 merges all indexed learning files (up to learning.maxFiles) using a memory-safe, one-file-at-a-time loop with time-decay weighting. See Section 7 for full details.

Output: output/application-map.json (built incrementally, page by page)

Stage 4 — Test Strategy

Prompt: prompts/04-test-strategy.prompt.md

Reads the application map and plans concrete test cases across smoke, regression, and e2e types. Maps each test case to the forms, buttons, and user journeys discovered in Stage 3.

Output: output/test-plan.json

Stage 5 — Data Factory

Prompt: prompts/05-data-factory.prompt.md

Generates test data builder classes (factories) for every form identified in Stage 3. Respects field types, validation rules, required flags, and the project's framework conventions.

Output: Data factory files written directly to project.existingProjectPath (or newProjectPath).

Stage 6 — POM Generator

Prompt: prompts/06-pom-generator.prompt.md

Generates Page Object Model classes for every page in the application map. Uses candidateLocators[] (populated by the locator-learning bias in Stage 3) to emit the most reliable locator first, with fallback chains in comments.

For Kendo UI and Angular Material controls, emits the correct interaction strategy (e.g., click-wrapper-then-list-item for Kendo dropdowns).

Output: POM files in src/pages/ (or framework-equivalent).

Stage 7 — Test Generator

Prompt: prompts/07-test-generator.prompt.md

Generates spec files from the test plan. Each test case becomes a test() / it() block with proper beforeEach, page navigation, form interaction via POM methods, assertion patterns, and teardown.

Output: Spec files in src/tests/ (or framework-equivalent).

Stage 8 — Gap Analyzer

Prompt: prompts/08-gap-analyzer.prompt.md

Compares the generated specs against the application map to identify coverage gaps: uncovered forms, untested conditional branches, missing error-state tests, etc. Emits a prioritized report.

Output: output/coverage-gap-report.json (kept after pipeline cleanup).

6. POM Locator Miner (Utility)

Prompt: prompts/utility-pom-locator-miner.prompt.md

A standalone utility (not part of the 8-stage pipeline) that scans your existing POM files and records every locator strategy in use. The output feeds Stage 3's locator-learning bias.

When to Run

SituationRun?
Onboarding an existing project to AutoTestGenYes
Major POM refactor (new conventions)Yes
New UI library introduced (Kendo / Material)Yes
Periodic refresh (e.g. monthly)Recommended
Pure greenfield project (no POMs yet)No

How to Run

  1. Ensure pipeline.config.json has project.existingProjectPath set (or pass an absolute path inline).
  2. In Copilot Chat, type: /utility-pom-locator-miner
  3. The agent scans POM files, aggregates locator strategies, and writes a timestamped JSON to learning/.
  4. After writing, the agent prunes learning/INDEX.json to at most learning.maxFiles entries (default 6).

Output

  • learning/locator-learning-<ISO-timestamp>.json — full scan result (one per run)
  • learning/INDEX.json — newest-first index of all scans (pruned to maxFiles)

What It Extracts

SectionContent
locatorStrategiesOverall % use of each strategy (data-testid, role, id, css, xpath, …)
byControlTypePreferred strategies per control type (button, input, dropdown, …)
byUiLibraryPreferred strategies per UI library (native, kendo, material)
interactionPatternsCommon helper method names to preserve in Stage 6
fallbackChainsMulti-locator fallback chains already in use by the team
recommendationsActionable suggestions (high / medium / low severity)
rawLocatorsPer-locator records (capped at rawLocatorsMax, default 1000)

7. Locator Learning System

The locator learning system is the mechanism that makes AutoTestGen smarter with each miner run. Instead of only using the most recent scan (which could miss patterns established earlier), Stage 3 now merges all indexed scan files into a single compact merged-bias object using a memory-safe, one-file-at-a-time loop.

Multi-File Merge Algorithm

  1. Read learning/INDEX.json (lightweight). Collect all scan entries, newest first. Cap at learning.maxFiles (default 6).
  2. For each file (one at a time, newest first): read the file → validate required keys → extract only aggregate sections (locatorStrategies, byControlType, byUiLibrary, interactionPatterns, fallbackChains). Skip rawLocators to avoid memory pressure.
  3. Assign a time-decay weight: newest = 1.0, next = 0.8, then 0.64, 0.512… This ensures recent conventions dominate while older runs still contribute signal.
  4. Merge the file's weighted data into the running mergedBias object (weighted counts, strategy votes).
  5. Release the file from memory immediately. Only mergedBias persists.
  6. Apply mergedBias to DOM capture — every field gets a candidateLocators[] ordered by the merged preferred strategies.

Merging Rules

SectionMerge Method
locatorStrategiesAccumulate weightedCount += count × weight. Recompute percentages at end. Top strategy by weighted count = topStrategy.
byControlTypeWeighted vote per strategy per control type (array index 0 = highest vote). Top 3 by vote score become merged preferredStrategies.
byUiLibrarySame voting as byControlType.
interactionPatternsUnion of all patterns; sum occurrences (weighted). Cap at 20.
fallbackChainsUnion of unique chains (dedup by fieldName). Cap at 50.

Memory Safety Guarantees

  • Only one learning file is open at any time — no bulk loading.
  • rawLocators (the largest section) is never loaded from any file.
  • The in-memory mergedBias object contains no raw locator strings or sample arrays — only aggregate statistics.
  • If a file fails validation, it is skipped with a warning — the crawl never fails due to a bad learning file.

INDEX Pruning

After each miner run, the miner prunes learning/INDEX.json to at most learning.maxFiles entries. Pruned entries are removed from the index (so Stage 3 doesn't process them) but their files remain on disk for manual archiving. 

# Archive files no longer in the index
ls -1t learning/locator-learning-*.json | tail -n +7 | \
  xargs -I {} mv {} learning/archive/

candidateLocators on Each Field

Stage 3 records the merged strategy ordering on every captured field:

"candidateLocators": [
  { "strategy": "data-testid", "selector": "[data-testid='submit-btn']", "source": "learning-bias" },
  { "strategy": "role",        "selector": "role=button[name='Submit']",  "source": "learning-bias" },
  { "strategy": "css",         "selector": "button.submit",               "source": "framework-fallback" }
]

Stage 6 uses the first candidate. If it breaks in CI, Stage 6 can rotate to the next without re-crawling.

8. Pipeline Modes

ModeConfigUse Case
full pipeline.mode: "full" First-time generation or full refresh across the entire application.
single-route pipeline.mode: "single-route"
pipeline.singleRoute: "/path"		 Generate or refresh artifacts for one new/changed route. Merges into the existing project — does not regenerate unchanged pages.

Single-Route Mode Details

When mode: "single-route":

  • Stage 3 skips the Route Discovery Pass entirely — crawls only the specified URL.
  • Stage 3 merges the new page into the existing application-map.json.
  • Stages 4–7 scope their work to forms/elements on that page and its direct modals/journeys.
  • Stage 8 scopes gap analysis to the single route's deltas.
  • pipeline-state.json uses stage3.singleRouteCompleted instead of the global stage3.status: "completed".

9. Resume & Checkpointing

Every stage writes its status to output/pipeline-state.json. If a run stalls (e.g., browser timeout, network error), re-invoking the master orchestrator automatically resumes from the last completed stage.

pipeline-state.json Key Fields

FieldDescription
stages.stage<N>.status"pending" | "in_progress" | "completed" | "failed"
stage3.lastCompletedRouteLast route fully written to application-map.json
stage3.routesCompletedCount of routes already captured

Stage 3 Resume Procedure

  1. Check output/pipeline-state.json for stage3.status.
  2. If "in_progress": load routesCompleted and rebuild visitedRoutes from the existing application-map.json. Drop those routes from crawlQueue.
  3. If a route has a partial entry (pageId exists but required fields missing), treat it as pending and re-capture.
  4. Log a RESUME banner: [RESUME] Resuming from route N of M.
Do not manually edit output/pipeline-state.json. Doing so may cause the resume logic to skip or re-process stages incorrectly.

11. Troubleshooting

SymptomLikely CauseFix
Stage 3 stalls / times out mid-crawl Large application — DOM accumulation in memory Re-run the orchestrator. Stage 3 resumes from the last completed route automatically.
"Playwright MCP server is not running" MCP server not started or npm install not run Run cd mcp-config/autotestgen-mcp && npm install once. VS Code will auto-start the MCP server from local node_modules/ when the chat session begins.
"Neither Google Chrome nor Microsoft Edge was found" No supported browser installed Install Google Chrome (preferred) or Microsoft Edge. The pipeline never downloads Chromium.
Stage 2 fails — login page not found Wrong AUTH_LOGIN_PAGE_TITLE or selector Update AUTH_USERNAME_SELECTOR, AUTH_PASSWORD_SELECTOR, AUTH_SUBMIT_SELECTOR in .env.
"No POMs found" in locator miner POMs not in standard folders Set projectFingerprint.folders.pages or pass the path inline at invocation.
Stage 3 not using learning files learning/INDEX.json missing or file not listed Re-run the miner. Check that the file passes basic schema validation.
byControlType is empty in learning file Field-name heuristics didn't match POM names Rename fields to follow *Button, *Input, *Dropdown conventions.
Kendo dropdown selection does nothing Wrong interaction strategy — using selectOption() on wrapper Stage 3's 3h must capture kendo-dropdownlist and emit interactionStrategy: "kendo-dropdown-click-list-item". Re-run Stage 3 after miner if needed.
pipeline-state.json shows wrong status Manual edit or interrupted write Delete output/pipeline-state.json and re-run from Stage 0. (Only do this if resume is not possible.)

12. Do's & Don'ts

DoDon't
Run the locator miner before every full pipeline run.Skip the miner on a project with existing POMs — you'll lose bias signal.
Use single-route mode for incremental updates.Re-run the full pipeline for a single new page — it's unnecessary.
Keep last 6 scan files in INDEX (default maxFiles).Let the index grow unbounded — it increases Stage 3 merge time.
Review recommendations from the miner in sprint planning.Treat miner recommendations as automatic tickets — they need human judgment.
Archive old learning files instead of deleting them.Hand-edit generated learning JSON files — the schema will break.
Set crawl.excludeRoutes for logout, error, and admin routes.Crawl /logout — it will destroy the auth session mid-crawl.
Use generation.mergeStrategy: "append-only" on existing projects.Use "overwrite" on an existing project — you'll lose custom locator tweaks.
Store secrets in .env only.Put secrets in pipeline.config.json directly.
